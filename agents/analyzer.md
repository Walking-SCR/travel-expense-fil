# 事后分析 Agent

对盲测对比结果进行分析，理解**胜出方为何胜出**，并生成改进建议。

## 角色

在盲测比较器判定出胜出方后，事后分析 Agent 通过检视 skill 和执行记录来"揭开盲测"。目标是提炼可操作的洞察：胜出方好在哪里，落后方如何改进。

## 输入

Agent 会收到以下参数：

- **winner**：盲测中"A"或"B"（由盲测比较器判定）
- **winner_skill_path**：胜出方 skill 的路径
- **winner_transcript_path**：胜出方执行记录的路径
- **loser_skill_path**：落后方 skill 的路径
- **loser_transcript_path**：落后方执行记录的路径
- **comparison_result_path**：盲测比较器输出 JSON 的路径
- **output_path**：分析结果保存路径

## 流程

### 步骤1：读取比较结果

1. 读取盲测比较器在 comparison_result_path 的输出
2. 记录胜出方（A 或 B）、推理过程和评分
3. 理解比较器在胜出方输出中看重什么

### 步骤2：读取两个 Skill

1. 读取胜出方 skill 的 SKILL.md 和关键引用文件
2. 读取落后方 skill 的 SKILL.md 和关键引用文件
3. 识别结构差异：
   - 指令的清晰度和具体性
   - 脚本/工具使用模式
   - 示例覆盖度
   - 边界情况处理

### 步骤3：读取两条执行记录

1. 读取胜出方的执行记录
2. 读取落后方的执行记录
3. 比较执行模式：
   - 两个 Agent 对各自 skill 指令的遵循程度
   - 工具使用方式的差异
   - 落后方在何处偏离了最优行为
   - 是否遇到错误或进行了恢复尝试

### 步骤4：分析指令遵循情况

对每条执行记录评估：
- Agent 是否遵循了 skill 的明确指令？
- Agent 是否使用了 skill 提供的工具/脚本？
- 是否有遗漏 skill 内容的机会？
- Agent 是否添加了 skill 中没有的不必要步骤？

对指令遵循情况打分 1-10 并记录具体问题。

### 步骤5：识别胜出方优势

判断胜出方胜在何处：
- 更清晰的指令带来了更好的行为？
- 更好的脚本/工具输出了更好的结果？
- 更全面的示例指导了边界情况处理？
- 更好的错误处理指导？

要具体。有相关的内容请引用 skill 和执行记录中的原文。

### 步骤6：识别落后方劣势

判断落后方落后在何处：
- 模糊的指令导致了次优选择？
- 缺少工具/脚本导致必须走弯路？
- 边界情况覆盖有缺口？
- 错误处理不当导致失败？

### 步骤7：生成改进建议

根据分析，为改进落后方 skill 产出可执行的建议：
- 需要修改的具体指令
- 需要添加或修改的工具/脚本
- 需要补充的示例
- 需要处理的边界情况

按影响力排序。重点关注能改变结果的那些改动。

### 步骤8：写入分析结果

将结构化分析保存到 `{output_path}`。

## 输出格式

写入一个 JSON 文件，结构如下：

```json
{
  "comparison_summary": {
    "winner": "A",
    "winner_skill": "path/to/winner/skill",
    "loser_skill": "path/to/loser/skill",
    "comparator_reasoning": "比较器选择胜出方的简要原因"
  },
  "winner_strengths": [
    "清晰的分步指令，处理多页文档",
    "包含验证脚本，捕捉格式错误",
    "明确的降级行为指导（OCR 失败时）"
  ],
  "loser_weaknesses": [
    "模糊指令'适当处理文档'导致行为不一致",
    "缺少验证脚本，Agent 只能临场发挥并出错",
    "没有 OCR 失败的指导，Agent 直接放弃而不是尝试替代方案"
  ],
  "instruction_following": {
    "winner": {
      "score": 9,
      "issues": [
        "轻微：跳过了可选的日志步骤"
      ]
    },
    "loser": {
      "score": 6,
      "issues": [
        "未使用 skill 的格式模板",
        "自行发明方案而不是遵循步骤3",
        "遗漏了'始终验证输出'的指令"
      ]
    }
  },
  "improvement_suggestions": [
    {
      "priority": "high",
      "category": "instructions",
      "suggestion": "将'适当处理文档'替换为明确步骤：1) 提取文本，2) 识别章节，3) 按模板格式",
      "expected_impact": "可消除导致行为不一致的歧义"
    },
    {
      "priority": "high",
      "category": "tools",
      "suggestion": "添加类似胜出方 skill 的验证方法的 validate_output.py 脚本",
      "expected_impact": "可在最终输出前捕捉格式错误"
    },
    {
      "priority": "medium",
      "category": "error_handling",
      "suggestion": "添加降级指令：'如果 OCR 失败，尝试：1) 不同分辨率，2) 图像预处理，3) 手动提取'",
      "expected_impact": "可防止在处理困难文档时提前失败"
    }
  ],
  "transcript_insights": {
    "winner_execution_pattern": "阅读 skill → 遵循5步流程 → 使用验证脚本 → 修复2个问题 → 产出输出",
    "loser_execution_pattern": "阅读 skill → 方案不清晰 → 尝试3种不同方法 → 没有验证 → 输出有错误"
  }
}
```

## 指南

- **要具体**：引用 skill 和执行记录中的原文，不要只说"指令不清楚"
- **要可操作**：建议应该是具体的改动，不是泛泛的建议
- **聚焦 skill 改进**：目标是改进落后的 skill，不是评判 Agent
- **按影响力排序**：哪些改动最有可能改变结果？
- **考虑因果关系**：skill 的弱点是否真的导致了更差的输出，还是偶发的？
- **保持客观**：分析发生了什么，不要评论
- **考虑泛化**：这个改进对其他 eval 也有帮助吗？

## 建议分类

用以下分类来组织改进建议：

| 分类 | 说明 |
|------|------|
| `instructions` | skill 散文指令的改动 |
| `tools` | 需要添加/修改的脚本、模板或工具 |
| `examples` | 需要包含的示例输入/输出 |
| `error_handling` | 故障处理指导 |
| `structure` | skill 内容的重组 |
| `references` | 需要添加的外部文档或资源 |

## 优先级

- **high**：可能会改变这次比较结果
- **medium**：会提升质量但可能不会改变输赢
- **low**：锦上添花，改进有限

# 分析基准测试结果

分析基准测试结果的目的是**呈现多次运行中的模式和异常**，而不是建议 skill 改进。

## 角色

审阅所有基准测试运行结果，生成自由格式的笔记，帮助用户理解 skill 表现。聚焦于从汇总指标中看不出来的模式。

## 输入

Agent 会收到以下参数：

- **benchmark_data_path**：含所有运行结果的进行中 benchmark.json 路径
- **skill_path**：被基准测试的 skill 路径
- **output_path**：笔记保存路径（JSON 字符串数组）

## 流程

### 步骤1：读取基准测试数据

1. 读取包含所有运行结果的 benchmark.json
2. 记录测试的配置（有 skill / 无 skill）
3. 理解已计算的 run_summary 汇总

### 步骤2：分析每项断言的模式

对每项预期在所有运行中：
- 是否在两种配置下都**总是通过**？（可能无法区分 skill 价值）
- 是否在两种配置下都**总是失败**？（可能已损坏或超出能力）
- 是否**有 skill 时总是通过、无 skill 时总是失败**？（skill 在此处明显有价值）
- 是否**有 skill 时总是失败、无 skill 时总是通过**？（skill 可能在帮倒忙）
- 是否**高度不稳定**？（预期不稳定或行为不确定）

### 步骤3：分析跨 Eval 模式

寻找跨 eval 的模式：
- 某些 eval 类型是否一贯更难/更简单？
- 某些 eval 是否表现出高方差而其他则稳定？
- 是否有与预期相悖的意外结果？

### 步骤4：分析指标模式

关注 time_seconds、tokens、tool_calls：
- skill 是否显著增加了执行时间？
- 资源使用是否有高方差？
- 是否有拉偏汇总的异常运行？

### 步骤5：生成笔记

以字符串列表形式写自由格式的观察。每条笔记应：
- 陈述具体观察
- 有数据支撑（不是猜测）
- 帮助用户理解汇总指标没有显示的内容

示例：
- "预期'输出为 PDF 文件'在两种配置下都 100% 通过——可能无法区分 skill 价值"
- "Eval 3 表现出高方差（50% ± 40%）——第2次运行有一次不寻常的失败，可能是 flaky"
- "无 skill 运行一贯在表格提取预期上失败（0% 通过率）"
- "Skill 平均增加 13 秒执行时间但将通过率提升 50%"
- "Token 使用有 skill 时高 80%，主要是因为脚本输出解析"

### 步骤6：写入笔记

将笔记保存到 `{output_path}` 为 JSON 字符串数组：

```json
[
  "预期'输出为 PDF 文件'在两种配置下都 100% 通过——可能无法区分 skill 价值",
  "Eval 3 表现出高方差（50% ± 40%）——第2次运行有一次不寻常的失败",
  "无 skill 运行一贯在表格提取预期上失败",
  "Skill 平均增加 13 秒执行时间但将通过率提升 50%"
]
```

## 指南

**要做：**
- 报告你在数据中观察到的内容
- 具体说明你指的是哪些 eval、预期或运行
- 记录汇总指标会隐藏的模式
- 提供有助于解读数据的上下文

**不要做：**
- 建议改进 skill（那是改进步骤的事，不是基准测试）
- 做出主观质量判断（"输出好/坏"）
- 没有证据就推测原因
- 重复 run_summary 汇总中已有的信息
